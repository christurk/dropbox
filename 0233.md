Proposal: Contract-Driven Platform Control Plane

We should build a golden-path platform where developers describe services using a high-level service contract, and the platform translates that contract into concrete artifacts for AKS, Azure, Auth0, Istio, observability, CI/CD, GitOps, and promotion.

The service contract should express developer intent, not low-level implementation details. For example, a developer should declare “this is an external API with Auth0 auth, Postgres, standard observability, and dev/staging/prod environments,” not raw Terraform variables, Helm values, Istio VirtualServices, or Azure resource definitions.

Example contract shape:

```yaml
apiVersion: platform.company.com/v1alpha1
kind: Service
metadata:
  name: payments-api
  owner: team-payments
spec:
  type: web-api
  runtime:
    language: dotnet
    containerPort: 8080
  environments:
    - name: dev
    - name: staging
    - name: prod
  exposure:
    type: external
    host: payments.company.com
    authRequired: true
  auth:
    provider: auth0
    audience: payments-api
    organizationMode: required
  dependencies:
    postgres:
      tier: standard
      size: small
  observability:
    tier: standard
    slo:
      availability: 99.9
      latencyP95Ms: 300
  delivery:
    promotion: dev-to-staging-to-prod
```

The key platform component is a translation engine, or platform compiler. It takes `service.yaml` and renders the desired downstream artifacts. It should be deterministic: same inputs produce the same outputs.

The platform compiler should generate things like:

* Rendered Kubernetes manifests for AKS
* Argo CD Applications and AppProjects
* Kargo Warehouses, Stages, and promotion config
* Istio/Gateway API routes
* Auth0 desired-state config
* OTEL collector/instrumentation config
* LGTM dashboards, alerts, labels, and SLO config
* Crossplane/ASO/Terraform inputs for Azure resources
* GitHub Actions workflow templates
* Backstage catalog metadata

The platform should not expose Terraform, Helm, Istio, Auth0, Crossplane, or ASO directly to developers. Those are implementation details behind the service contract.

Recommended architecture:

```text
Backstage / CLI / Agent
        ↓
service.yaml PR
        ↓
service contract registry
        ↓
validation + policy + semantic diff
        ↓
platform compiler
        ↓
generated artifacts
        ↓
Kargo + Argo CD + IaC reconcilers
        ↓
AKS, Azure, Auth0, Istio, LGTM
```

Use Git as the control boundary. Keep a contract registry repo for service definitions and a rendered manifests repo or rendered branches for deployable output.

Recommended repo model:

```text
platform-service-registry/
  services/payments-api/service.yaml
  schemas/
  policies/

gitops-manifests/
  env/dev/services/payments-api/
  env/staging/services/payments-api/
  env/prod/services/payments-api/
```

For the rendered manifest pattern, Argo CD workload apps should point to environment-specific rendered output, not raw source `main`.

Example:

```text
main:
  service contracts, templates, platform source, Kargo definitions

env/dev:
  rendered dev manifests

env/staging:
  rendered staging manifests

env/prod:
  rendered prod manifests
```

GitHub Actions should build immutable app artifacts but should not own deployment or promotion.

GitHub Actions responsibilities:

* test
* lint
* build image
* scan image
* sign/attest
* create SBOM
* push image by digest
* validate service contract and render preview

Kargo responsibilities:

* detect promotable image digests
* promote release bundles through dev, staging, prod
* render environment-specific manifests
* commit rendered output to environment branches/paths
* coordinate with Argo CD

Argo CD responsibilities:

* sync rendered manifests to AKS
* report health and drift
* enforce GitOps deployment state

A release should be treated as a bundle, not just an image tag.

Example release bundle:

```yaml
service: payments-api
imageDigest: sha256:abc123
contractRevision: 8f31a2c
platformProfile: web-api
platformProfileVersion: 1.8.2
envConfigRevision: 39a21dd
```

Kargo should promote this bundle. That means rendering happens during promotion. Even if only the image digest changes, Kargo re-renders the manifests and commits the result. This is expected and useful because Argo CD always deploys fully rendered desired state.

Service contract changes may also need promotion. Some contract changes only alter app manifests, such as CPU/memory, routes, OTEL config, or environment variables. These should move through dev, staging, and prod like a release.

Other contract changes are infrastructure lifecycle changes, such as database size, storage, tenant removal, Auth0 changes, or external exposure. These need stricter controls: semantic diffing, policy checks, generated plans, approval gates, and deletion protection.

The platform compiler should classify changes:

```text
safe
risky
destructive
security-sensitive
cost-sensitive
production-gated
forbidden
```

Examples:

```text
replicas 2 → 3:
  low risk

internal route → external route:
  security-sensitive

authRequired true → false:
  forbidden or security exception required

Postgres small → large:
  cost-sensitive and production-gated

delete database:
  destructive and blocked by default
```

Every service-contract PR should produce a generated preview:

```text
What changed in the contract
What rendered manifests changed
What infra resources are affected
Which environments are affected
Risk classification
Required approvals
```

For production safety, the platform should enforce:

* policy-as-code in CI
* admission control in clusters
* environment promotion gates
* field-level ownership
* production approval for sensitive fields
* deletion protection for databases/storage/Auth0/DNS/tenant resources
* generated manifest and infrastructure diffs before merge
* pinned platform profile versions

Platform profiles should be versioned. For example:

```yaml
platform:
  profile: web-api
  version: 1.8.2
```

This avoids changing every production service when the platform team updates a base template. Services can upgrade from `web-api 1.8.2` to `web-api 1.9.0` through a controlled promotion path.

For IaC, use a hybrid model:

* Crossplane and/or ASO for service-scoped Azure resources
* Terraform for global/shared resources, Auth0, GitHub configuration, SaaS providers, and gaps
* Do not make developers choose between them

The likely implementation path:

1. Define the `Service` contract schema.
2. Create a contract registry repo.
3. Build a simple `platformctl render` CLI that turns `service.yaml` into artifacts.
4. Run validation and render preview in CI.
5. Generate rendered Kubernetes manifests, Argo CD Applications, Kargo config, Auth0 config, OTEL config, and IaC inputs.
6. Have Kargo promote release bundles and render environment-specific manifests.
7. Have Argo CD sync only rendered environment output.
8. Add policy checks, semantic diffing, approval gates, and deletion protection.
9. Put Backstage on top as the developer UI.
10. Add agentic workflows that propose service contract changes and open PRs.
11. Only later consider a Kubernetes operator if we need live reconciliation/status for `ServiceContract` resources.

We probably do not need to start by writing a Kubernetes operator. Start with a deterministic platform compiler plus Git workflow. An operator becomes useful later if the service contract itself needs to be a live Kubernetes resource with status, reconciliation, events, and continuous drift handling.

There is no single off-the-shelf tool that solves the entire problem. Existing tools cover slices:

* Backstage: portal and developer workflows
* Argo CD: GitOps reconciliation
* Kargo: promotion
* Crossplane/ASO/Terraform: infrastructure reconciliation
* OPA/Kyverno: policy
* Score/Kratix: possible inspiration or building blocks for contract-driven platform APIs

The custom part we likely need is the platform-specific compiler and policy/diff engine that understands our stack: Auth0, multi-tenant environments, AKS, Azure, Istio, OTEL/LGTM, GitHub Actions, Kargo, Argo CD, Backstage, and agent workflows.

The main design principle:

Developers own intent.
The platform owns implementation.
Promotion owns environment rollout.
Reconcilers own actual state.
